### SOP #27 — editable install drift 永久教训
1. editable install 必须指源仓, 不能指 `~/.openclaw/<repo>/` 双份
2. 看 `pip show <pkg>` 的 `Editable project location:` 确认指向
3. 修改源仓后不需要 sync, 立即生效
4. `python3 -c "import <pkg>; print(<pkg>.__file__)"` 是不靠 cwd 的最简验证
5. drift 检测: `diff <(cd src && find . -name '*.py' | sort) <(cd ~/.openclaw/... && find . -name '*.py' | sort)` 或 `md5sum` 关键文件

## 07:40 SOP #28 V verify 分层 (10 分钟)

**问题**: Hermes 报告 "18/18 通过" 实测 13/13 (SOP #25), 浮光/外部报告不可全信. 但每次都全跑 e2e 太重.

**解法**: L1/L2 分层 verify

| Tier | 范围 | 工具 | 时延 |
|------|------|------|------|
| L1 | 端口/文件/commit SHA/状态码 | grep/curl/pip show | <5s |
| L2 | 端到端功能/性能/集成行为 | pytest e2e / watchdog / 人工 | 30s-2min |

**规则**:
- 任何外部报告 (Hermes/Human/其他 agent) 数字一律不采信
- 报告里所有"事实" (X 文件存在 / Y 端口在听 / Z commit 已推) 走 L1 复核
- 报告里所有"功能" (test passed / benchmark 数字 / 集成工作) 走 L2 自跑
- 自跑可以 sample (关键 case 5-10 个) 不用全跑, 但要覆盖报告里所有声称的 P0/P1 能力
- 数字 < 报告里说的 = 报告失实, 立 SOP 立碑

### SOP #28 — V verify 分层
1. 外部报告数字不可信, 走 L2 自跑 sample
2. 外部报告事实可走 L1 grep/curl 复核
3. 自跑 sample 必覆盖报告所有 P0/P1 声称
4. 数字 < 报告 = 失实, 立碑
5. 跟 SOP #25 (Hermes) 配套: #25 是案例, #28 是规则
